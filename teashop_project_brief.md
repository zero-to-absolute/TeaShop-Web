# TeaShop — Подготовительная информация к разработке

> Этот документ содержит предварительное описание проекта, собранное на этапе
> анализа и планирования. Часть деталей (особенно по БД) будет уточнена
> непосредственно в процессе разработки.

---

## 1. О проекте

**Тема:** Разработка информационной системы с конструктором персонализированного
подбора состава для чайного магазина.

**Контекст:** Учебный дипломный проект. Для целей разработки используется легенда
вымышленного предприятия — специализированный чайный магазин **«TeaShop»**,
работающий как ИП. В рамках стажировки поставлена задача разработки
информационной системы с функционалом онлайн-витрины и персонализированного
подбора чайных смесей.

---

## 2. Суть системы

Веб-приложение состоит из двух ключевых блоков:

1. **Интернет-витрина** — каталог чаёв и ингредиентов с фильтрацией и карточками товаров
2. **Персонализированный конструктор** — пользователь проходит анкету
   (вкус, аромат, желаемый эффект) → система подбирает базовый чай →
   предлагает совместимые ингредиенты пошагово → формирует итоговый состав

Дополнительно:
- Система рекомендаций похожих готовых составов по результатам подбора
- Логика совместимости: каждый чай и каждый ингредиент имеют связи
  совместимости между собой
- Корзина для авторизованных пользователей
- Выбор формата продажи: готовая смесь или компоненты раздельно

## 3. Технологический стек

| Слой | Технология |
|---|---|
| Разметка | HTML5 |
| Стили | CSS3 (без фреймворков) |
| Клиентская логика | JavaScript ES6+ (без фреймворков) |
| База данных | PostgreSQL через Supabase |
| Аутентификация | Supabase Auth |
| Хранилище файлов | Supabase Storage (фото товаров) |
| Безопасность | Row Level Security (RLS) в PostgreSQL |
| Стилизация (внешняя) | AI Studio / Gemini |
| Изображения (внешняя) | AI Studio / Imagen |
| Администрирование каталога | WPF-приложение (внешнее, уже существует) |

**Важно:** бэкенд отсутствует намеренно. Вся логика выполняется на клиенте
(JS в браузере), безопасность обеспечивается через RLS на уровне БД.

## 4. Архитектура

**Тип:** JAMstack (JavaScript + API + Markup) без серверного уровня.

```
Браузер (HTML/CSS/JS)
        │
        │  Supabase JS SDK
        ▼
    Supabase
    ├── Auth (auth.users)
    ├── PostgreSQL (public.*)
    └── Storage (фото)
        ▲
        │  Supabase JS SDK
WPF-приложение (внешнее, управление каталогом)
```

**Почему без бэкенда:**
- Объём каталога и сложность алгоритма рекомендаций не требуют серверной обработки
- RLS в PostgreSQL заменяет серверную авторизацию
- Supabase Auth закрывает аутентификацию
- Алгоритм scoring совместимости реализуется на JS (каталог в пределах сотен позиций)

**Перспектива масштабирования:** при усложнении логики рекомендаций до ML
или добавлении оплаты — возможно введение серверного уровня без изменения
структуры БД и фронтенда.

## 5. Пользователи системы

Система поддерживает два режима:

| Режим | Доступные функции |
|---|---|
| **Гость** (без регистрации) | Каталог, фильтрация, карточки товаров, конструктор |
| **Авторизованный пользователь** | Всё выше + корзина, сохранение составов |

**Авторизация:** email + пароль через Supabase Auth.

**Паттерн пользовательских данных:**
- `auth.users` — системная таблица Supabase (UUID, email, хэш пароля)
- `public.profiles` — пользовательская таблица (UUID как FK, имя, аватар и т.д.)
- Связь через триггер PostgreSQL: при регистрации автоматически создаётся
  запись в `public.profiles`

## 6. Функции системы

### 6.1. Каталог
- Список чаёв: наименование, описание, характеристики (вкус, аромат, эффект), цена
- Список ингредиентов: характеристики, совместимость
- Фильтрация и поиск по категории и вкусовому профилю
- Детальная карточка товара

### 6.2. Конструктор
- Пошаговая анкета: вкусовые предпочтения, аромат, желаемый эффект
- Подбор базового чая по ответам анкеты
- Пошаговые рекомендации совместимых ингредиентов прямо в процессе сборки
  (выбрал чай → предлагается компонент → выбрал компонент → предлагается следующий)
- Формирование итогового состава
- Выбор формата продажи: **готовая смесь** (всё вместе) или **компоненты раздельно**
- Индикатор прогресса по шагам конструктора

### 6.3. Система рекомендаций
- Контекстные рекомендации на каждом шаге сборки (на основе таблицы совместимости)
- Степень совместимости — вычисление соответствия итогового состава с готовыми рецептами
- Похожие готовые составы по завершении конструктора

### 6.4. Корзина
- Добавление товара из каталога (только авторизованный пользователь)
- Добавление результата конструктора в корзину
- Кнопка «Оформить заказ» — отображается, но **неактивна** (задел на будущее)
- Неавторизованному пользователю при попытке добавить — предложение войти/зарегистрироваться

### 6.5. Администрирование
- Управление каталогом осуществляется через **внешнее WPF-приложение**,
  работающее напрямую с той же БД Supabase
- Отдельный административный веб-интерфейс **не разрабатывается**

## 7. Ключевые требования

### Функциональные (сокращённо)
- Каталог с фильтрацией и карточками товаров
- Конструктор с анкетой, пошаговыми рекомендациями и итоговым составом
- Система рекомендаций похожих составов
- Авторизация через Supabase Auth
- Корзина для авторизованных пользователей
- Выбор формата (смесь / раздельно)

### Нефункциональные (ключевые)
- Загрузка страницы каталога ≤ 3 сек
- Результат конструктора ≤ 2 сек после анкеты
- Адаптивный дизайн от 375px
- Совместимость: Chrome, Firefox, Edge (актуальные версии)
- RLS: данные корзины и профиля доступны только владельцу
- Anon key Supabase — только чтение публичных данных каталога
- Состояние анкеты конструктора сохраняется локально (не теряется при потере соединения)
- Пагинация или ленивая загрузка каталога
- Структура кода разбита по модулям: каталог, конструктор, корзина, авторизация

## 8. Что НЕ входит в проект

- Бэкенд / серверный уровень
- Оплата заказов
- Административный веб-интерфейс
- Мобильное приложение
- ML-модель рекомендаций (только JS-алгоритм на основе таблицы совместимости)
- Система уведомлений / email-рассылок

## 10. Заметки и открытые вопросы

- Детальная схема БД (таблицы, связи, поля) будет определена на этапе
  разработки отдельно — здесь не зафиксирована намеренно
- Конкретный алгоритм scoring совместимости (формула/логика) подлежит уточнению и проработки
- Supabase Realtime для каталога не требуется (данные меняются редко);
  актуально только для корзины при необходимости
- Фото товаров будут хранится в Supabase Storage, ссылки — в таблице товаров
