### ШАГ 0 — Подготовка контекста

Прочитай и сохрани в память для передачи агентам:
1. Указанный файл фазы плана (`$ARGUMENTS`)
2. Соответствующий design-документ из `docs/design/`
3. `CLAUDE.md` — стандарты проекта
4. Существующие файлы, которые будут изменены (если есть)

---

### ШАГ 1 — Агент КОДЕР

Запусти суб-агент с типом `general-purpose` со следующим заданием:

```
ТЫ — КОДЕР. Реализуй фазу строго по плану.

ПЛАН ФАЗЫ:
[вставь полное содержимое файла фазы]

DESIGN-ДОКУМЕНТ:
[вставь ключевые части: компонентная схема, data flow, sequence]

СТАНДАРТЫ КОДА (обязательно):
- ES modules (import/export) везде
- Обработка ошибок для КАЖДОГО Supabase-запроса (try/catch)
- Проверка auth.uid() перед операциями с корзиной
- Комментарии на русском языке
- Имена переменных и функций на английском
- Адаптивность от 375px (CSS)
- Ключи Supabase только в src/js/shared/config.js — нигде больше

ПРАВИЛА:
- Пиши код только для текущей фазы, не забегай вперёд
- Создай или измени только файлы, указанные в плане
- Не добавляй серверную логику (проект без бэкенда)
- Не используй фреймворки (только ванильный JS/HTML/CSS)

Напиши код, создай/измени файлы. По завершении — выведи список
созданных/изменённых файлов с кратким описанием каждого.
```

Дождись завершения Кодера. Сохрани список изменённых файлов.

### ШАГ 2 — Агент РЕВЬЮЕР

Запусти суб-агент с типом `general-purpose` со следующим заданием:

```
ТЫ — РЕВЬЮЕР. Проверь написанный код.

ФАЙЛЫ ДЛЯ ПРОВЕРКИ:
[вставь список файлов от Кодера — ревьюер должен их прочитать]

ПЛАН ФАЗЫ (что должно быть реализовано):
[вставь содержимое файла фазы]

ПРОВЕРЬ ПО КАЖДОМУ ПУНКТУ и дай статус ✅ / ❌ / ⚠️:

БЕЗОПАСНОСТЬ:
- [ ] Нет XSS-уязвимостей (innerHTML с пользовательскими данными)
- [ ] Нет SQL-инъекций (все запросы через Supabase SDK)
- [ ] Нет захардкоженных ключей, URL, паролей в JS-файлах
- [ ] auth.uid() проверяется перед операциями с корзиной/профилем

КАЧЕСТВО КОДА:
- [ ] Все Supabase-запросы обёрнуты в try/catch
- [ ] ES modules (import/export) — нет глобальных переменных
- [ ] Имена функций/переменных на английском, комментарии на русском
- [ ] Нет серверной логики (Edge Functions, fetch к своему API)
- [ ] Нет фреймворков (React, Vue и т.д.)

СООТВЕТСТВИЕ ПЛАНУ:
- [ ] Реализовано всё из плана фазы
- [ ] Только то, что в плане (не больше)
- [ ] Структура файлов соответствует архитектуре проекта

АДАПТИВНОСТЬ:
- [ ] CSS работает от 375px (если фаза затрагивает стили)

Если есть ❌ — укажи конкретно: файл, строка, проблема, как исправить.
Если всё ✅ — напиши "КОД ПРОШЁЛ РЕВЬЮ".
```

Дождись результата Ревьюера.

**Если есть ❌:** верни Кодеру конкретные замечания (повтори Шаг 1 с уточнениями).
**Если всё ✅:** переходи к Шагу 3.

### ШАГ 3 — Агент ТЕСТИРОВЩИК

Запусти суб-агент с типом `general-purpose` со следующим заданием:

```
ТЫ — ТЕСТИРОВЩИК. Составь инструкцию ручной проверки.

РЕАЛИЗОВАННАЯ ФАЗА:
[вставь содержимое файла плана — что было реализовано]

ИЗМЕНЁННЫЕ ФАЙЛЫ:
[вставь список файлов от Кодера]

СОСТАВЬ инструкцию по ручному тестированию в браузере:

Формат каждого теста:
**Тест N: [название]**
- Условие: [что нужно сделать перед тестом]
- Шаги: [пронумерованный список действий]
- Ожидаемый результат: [что должно произойти]
- Как проверить в консоли: [если нужно — что смотреть в DevTools]

ОБЯЗАТЕЛЬНО проверь:
- Основной happy path (всё работает правильно)
- Edge cases (пустые поля, неверные данные, нет сети)
- RLS: данные одного пользователя не видны другому
  (если фаза затрагивает авторизованные операции)
- Адаптивность: уменьши окно до 375px — всё работает?
- Консоль браузера: нет красных ошибок

Выведи готовую инструкцию тестирования.
```
