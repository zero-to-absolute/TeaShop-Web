# TeaShop — Инженерный процесс AI-разработки

> Этот документ описывает **процесс работы с AI-ассистентом** (Claude) при
> разработке проекта TeaShop. Он дополняет `teashop_project_brief.md` и задаёт
> структуру, роли и правила, по которым AI должен вести разработку.
>
> **Цель:** получить качественный, поддерживаемый код вместо хаотичной генерации.

---

## 1. Философия: контекстная инженерия

AI генерирует код ровно такого качества, какой контекст ему передан.

**Формула качества:**

```
Качество = (Корректность + Полнота) / (Размер контекста + Шум)
```

Это значит:
- Передавай AI **только релевантную** информацию для текущего шага
- Не смешивай задачи — каждая фаза имеет **свой промпт и свой контекст**
- Факты отдельно, решения отдельно, код отдельно
- Контекстное окно — ограниченный ресурс; не засоряй его

---

## 2. Четыре фазы разработки каждой фичи

Каждая задача (новая фича, баг, улучшение) проходит строго 4 фазы.
**AI пишет код только на фазе 4.** Всё остальное — подготовка контекста.

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────────┐
│ RESEARCH │ ──► │  DESIGN  │ ──► │   PLAN   │ ──► │  IMPLEMENT   │
│ (as-is)  │     │ (to-be)  │     │ (шаги)   │     │ (код + тест) │
└──────────┘     └──────────┘     └──────────┘     └──────────────┘
     │                │                │                    │
  Документ         Диаграммы        Файлы фаз        Рабочий код
  фактов           + ADR           01_...md          + тесты
```

### Правило перехода между фазами:
- Результат каждой фазы **ревьюируется человеком** перед переходом к следующей
- Если что-то не так — проще исправить на текущей фазе, чем перегенерировать код
- Между фазами делай `/clear` (или начинай новый чат), чтобы сбросить контекст

---

## 3. Фаза 1: RESEARCH (Исследование)

### Цель
Сузить область проекта до набора фактов, относящихся к конкретной задаче.

### Что делает AI
- Изучает структуру проекта, файлы, функции
- Находит все релевантные модули, зависимости, паттерны
- Создаёт **документ фактов** (что, где, как устроено)

### Правила для AI на этой фазе

```
ТЫ НАХОДИШЬСЯ В ФАЗЕ RESEARCH.

ТВОЯ ЗАДАЧА:
- Изучи структуру проекта и найди всё, что относится к задаче: [ОПИСАНИЕ ЗАДАЧИ]
- Создай документ research с фактами о текущем состоянии проекта

ФОРМАТ ДОКУМЕНТА:
Только факты. Никаких советов, мнений, рекомендаций.
Не пиши "тут бы надо отрефакторить" или "здесь можно улучшить".
Описывай только то, что ЕСТЬ сейчас (as-is):
- Какие файлы затронуты
- Какие функции/модули связаны с задачей
- Какие паттерны уже используются в проекте
- Какие зависимости и интеграции существуют (Supabase SDK, RLS-политики и т.д.)
- Какие таблицы БД связаны с задачей

НЕ ПИШИ КОД. НЕ ПРЕДЛАГАЙ РЕШЕНИЯ.
```

### Выходной артефакт
Файл `research_[название-задачи].md` — сжатая карта релевантной части проекта.

### Адаптация под TeaShop
Проект использует JAMstack без бэкенда. При research особое внимание:
- Структура JS-модулей (каталог, конструктор, корзина, авторизация)
- RLS-политики в Supabase (какие таблицы, какие правила)
- Supabase SDK: какие методы уже используются
- Существующие паттерны взаимодействия с API
- HTML-структура страниц и маршрутизация

---

## 4. Фаза 2: DESIGN (Проектирование)

### Цель
Создать архитектурное решение задачи в виде визуальных диаграмм и описаний.

### Что делает AI
На вход получает: research-документ + описание задачи + этот файл (стандарты).
Создаёт архитектурное решение.

### Правила для AI на этой фазе

```
ТЫ НАХОДИШЬСЯ В ФАЗЕ DESIGN.

ВХОДНЫЕ ДАННЫЕ:
- Research-документ: [ССЫЛКА НА ФАЙЛ]
- Задача: [ОПИСАНИЕ]
- Стек: HTML5 + CSS3 + JS ES6+ + Supabase (PostgreSQL, Auth, Storage)
- Архитектура: JAMstack, без бэкенда, RLS для безопасности

СОЗДАЙ:
1. Диаграмму компонентов (какие JS-модули участвуют, как связаны)
2. Data flow диаграмму (откуда данные → куда идут → что возвращается)
3. Sequence диаграмму (порядок взаимодействий: пользователь → JS → Supabase)
4. Описание изменений в схеме БД (если нужны новые таблицы/поля/RLS)
5. ADR (Architecture Decision Record) — если есть выбор между подходами

Диаграммы создавай в формате Mermaid (встроенные в markdown).

НЕ ПИШИ КОД. Только архитектура и дизайн.
```

### Выходной артефакт
Файл `design_[название-задачи].md` — диаграммы + ADR + описание изменений в БД.

### Ревью человеком
Перед переходом к планированию ответь себе:
- Логична ли последовательность действий на sequence-диаграмме?
- Не нарушена ли модульность (каталог, конструктор, корзина, авторизация)?
- Учтены ли RLS-политики для новых таблиц/полей?
- Не добавляется ли серверная логика (проект без бэкенда)?

## 5. Фаза 3: PLAN (Планирование)

### Цель
Разбить дизайн на **последовательные, завершённые фазы реализации**.

### Что делает AI
На вход получает: утверждённый design + research.
Создаёт набор файлов-планов.

### Правила для AI на этой фазе

```
ТЫ НАХОДИШЬСЯ В ФАЗЕ PLAN.

ВХОДНЫЕ ДАННЫЕ:
- Design-документ: [ССЫЛКА]
- Research-документ: [ССЫЛКА]

СОЗДАЙ НАБОР ФАЙЛОВ ПЛАНА:
Каждый файл = одна завершённая фаза реализации.
Формат имён: 01_[название].md, 02_[название].md, ...

КАЖДАЯ ФАЗА ДОЛЖНА БЫТЬ:
✅ Завершённой — после неё проект компилируется/работает
✅ Тестируемой — можно проверить результат
✅ Атомарной — один коммит, одна логическая единица
✅ Последовательной — каждая фаза опирается на предыдущую

СОДЕРЖИМОЕ КАЖДОГО ФАЙЛА ФАЗЫ:
- Что именно реализуется
- Какие файлы создаются / изменяются
- Какие функции / классы / модули затрагиваются
- Критерии приёмки (как проверить, что фаза завершена)
- Какие тесты должны быть написаны

НЕ ПИШИ КОД. Только план.
```

### Выходной артефакт
Папка `plan_[название-задачи]/` с файлами `01_*.md`, `02_*.md`, ...

### Пример для TeaShop (задача: конструктор)
```
plan_constructor/
├── 01_data_model.md          — таблицы совместимости в Supabase
├── 02_supabase_queries.md    — JS-функции запросов к БД
├── 03_quiz_ui.md             — HTML/JS анкеты (шаги 1-3)
├── 04_scoring_algorithm.md   — алгоритм подбора чая
├── 05_ingredient_steps.md    — пошаговый подбор ингредиентов
├── 06_result_page.md         — итоговая страница состава
└── 07_integration_cart.md    — связка с корзиной
```

## 6. Фаза 4: IMPLEMENT (Реализация)

### Цель
Написать код по утверждённому плану с проверкой качества.

### Роли AI-агентов
Реализация ведётся как работа команды. Даже если это один чат — мысленно
разделяй роли:

| Роль | Обязанности |
|---|---|
| **Кодер** | Пишет код строго по текущей фазе плана |
| **Ревьюер** | Проверяет код: качество, безопасность, соответствие дизайну |
| **Тестировщик** | Пишет и запускает тесты, сообщает о проблемах |

### Правила для AI на этой фазе

```
ТЫ НАХОДИШЬСЯ В ФАЗЕ IMPLEMENT.

ВХОДНЫЕ ДАННЫЕ:
- Файл текущей фазы плана: [ССЫЛКА НА 0X_*.md]
- Design-документ: [ССЫЛКА]
- Существующий код проекта

РЕАЛИЗУЙ СТРОГО ПО ПЛАНУ:
1. Пиши код только для текущей фазы (не забегай вперёд)
2. После написания кода — проверь его как ревьюер:
   - Соответствует ли дизайну?
   - Нет ли проблем безопасности (XSS, инъекции, утечки ключей)?
   - Следует ли существующим паттернам проекта?
   - Нет ли захардкоженных данных?
3. Напиши тесты для написанного кода
4. Проверь, что всё работает

СТЕК ПРОЕКТА:
- HTML5 + CSS3 (без фреймворков)
- JavaScript ES6+ (без фреймворков), модульная структура
- Supabase JS SDK для работы с БД, Auth, Storage
- PostgreSQL с RLS для безопасности

СТАНДАРТЫ КОДА:
- ES modules (import/export)
- Модульная структура: каталог, конструктор, корзина, авторизация
- Комментарии на русском языке
- Осмысленные имена переменных и функций (на английском)
- Обработка ошибок для всех Supabase-запросов
- Проверка auth-состояния перед операциями с корзиной
```

### Quality Gates (гейты качества)
Фаза считается завершённой ТОЛЬКО если:

- [ ] Код соответствует плану текущей фазы
- [ ] Нет ошибок в консоли браузера
- [ ] Тесты написаны и проходят
- [ ] RLS-политики учтены (данные доступны только владельцу)
- [ ] Нет захардкоженных ключей, паролей, URL
- [ ] Код следует модульной структуре проекта
- [ ] Адаптивность сохранена (от 375px)

Если хоть один гейт не пройден — фаза не считается завершённой.
Ревьюер возвращает **конкретную проблему** (файл, строка, суть) кодеру.

## 7. Специализация промптов по типу задачи

Один универсальный промпт на всё — плохо. Разные задачи требуют разного контекста.

| Тип задачи | Фокус Research | Фокус Design | Фокус Plan |
|---|---|---|---|
| **Новая фича** | Архитектура, паттерны, связи | C4 + data flow + sequence | Фазы реализации по слоям |
| **Баг** | Воспроизведение, причины, затронутые файлы | Минимальное исправление | Одна фаза: фикс + тест |
| **Рефакторинг** | Текущая структура, зависимости | Целевая структура | Пошаговая миграция |
| **БД** | Существующие таблицы, RLS, связи | ERD + миграции | DDL-скрипты по шагам |

## 8. Управление контекстом в чате

### Принципы
- **Одна задача = один чат** (или `/clear` между задачами)
- **Одна фаза = один чат** (research отдельно, design отдельно, и т.д.)
- Если AI ошибся 2+ раза подряд — `/clear` и перепиши промпт лучше
- Не вставляй весь проект в контекст — только релевантные файлы

### Что передавать на каждой фазе

| Фаза | Передавай в контекст | НЕ передавай |
|---|---|---|
| Research | Задачу + путь к проекту | Мнения, решения |
| Design | Research-документ + задачу + стандарты | Весь код проекта |
| Plan | Design + Research | Код, который ещё не написан |
| Implement | Текущий файл плана + Design + код | Все файлы плана сразу |

### Для Claude Code / Claude Chat
- Используй `CLAUDE.md` в корне проекта для постоянного контекста (стек, стандарты кода)
- Для каждого типа задачи создай отдельный skill в `.claude/skills/`
- Subagents — для исследования кодовой базы (не засоряют основной контекст)

## 10. Шаблон CLAUDE.md для TeaShop

Положи этот файл в корень проекта. Он загружается в каждый чат автоматически.

```markdown
# TeaShop — Контекст проекта

## Стек
- HTML5 + CSS3 (без фреймворков)
- JavaScript ES6+ (без фреймворков)
- Supabase: PostgreSQL + Auth + Storage
- Безопасность через RLS

## Архитектура
- JAMstack без бэкенда
- Вся логика на клиенте (JS в браузере)
- Модули: каталог, конструктор, корзина, авторизация

## Стандарты кода
- ES modules (import/export)
- Обработка ошибок для КАЖДОГО Supabase-запроса
- Проверка auth перед операциями с корзиной
- Комментарии на русском
- Имена переменных/функций на английском
- Адаптивный дизайн от 375px

## Что НЕ делать
- НЕ добавлять серверный уровень
- НЕ использовать фреймворки (React, Vue и т.д.)
- НЕ хардкодить ключи/URL Supabase в JS-файлах (выноси в config)
- НЕ игнорировать RLS — каждая новая таблица должна иметь политики

## Тестирование
- Ручная проверка в браузере (Chrome, Firefox, Edge)
- Консоль без ошибок = базовый критерий
- Проверка RLS: данные одного пользователя не видны другому
```

## 11. Быстрый старт: как начать работу над задачей

### Шаг 1 — Сформулируй задачу
Чётко и конкретно. Не «сделай каталог», а:
> «Реализовать страницу каталога чаёв: список карточек с фильтрацией
> по категории и вкусовому профилю, пагинация по 12 элементов,
> данные из таблицы `teas` в Supabase»

### Шаг 2 — Research (новый чат)
Скопируй промпт из раздела 3, подставь свою задачу. Получи документ фактов.
Прочитай его. Всё верно? Переходи дальше.

### Шаг 3 — Design (новый чат)
Скопируй промпт из раздела 4, приложи research-документ. Получи диаграммы.
Проверь логику. Всё ок? Дальше.

### Шаг 4 — Plan (новый чат)
Скопируй промпт из раздела 5, приложи design + research. Получи файлы фаз.
Проверь, что каждая фаза атомарна и тестируема. Дальше.

### Шаг 5 — Implement (новый чат на КАЖДУЮ фазу плана)
Скопируй промпт из раздела 6. Передай ОДИН файл фазы + design.
Получи код. Проверь quality gates. Закоммить. Следующая фаза.

## 12. Ключевые принципы (памятка)

1. **AI пишет код только на одном этапе (Implement).** Всё остальное — подготовка контекста.
2. **Каждая фаза — свой чат.** Не смешивай research и код в одном окне.
3. **Человек ревьюирует после каждой фазы.** AI не принимает решения за тебя.
4. **Промпты специализированы.** Фича ≠ баг ≠ рефакторинг.
5. **Research — только факты.** Без мнений, без советов, без кода.
6. **Quality gates обязательны.** Нет зелёных тестов — нет коммита.
7. **Контекст — ценный ресурс.** Передавай только то, что нужно для текущего шага.
